================================================================
  CHANGES MADE TO THE FORMS PROJECT — WHAT, WHY, HOW TO AVOID

----------------------------------------------------------------
CHANGE 1 — dialog closes on every tab save
FILE: demo-dialog.component.ts — saveFullVehicle()
----------------------------------------------------------------

WHAT WAS WRONG:
  Every tab (Administration, SIM, Carburant, etc.) called saveFullVehicle().
  That method always called this.dialogRef.close(updatedVehicle) at the end.
  So clicking "Enregistrer" on any tab other than Tab 0 closed the entire dialog,
  preventing the user from filling other tabs in the same session.
  Tab 0 (save()) correctly did NOT close — this was inconsistent.

WHAT WAS FIXED:
  Removed this.dialogRef.close() from saveFullVehicle().
  Now all tabs save silently and show a snackbar confirmation instead.
  The dialog stays open so the user can continue to other tabs.
  The user closes the dialog manually via the X button in the header.

HOW TO AVOID IT NEXT TIME:
  In multi-tab dialogs, never auto-close on save unless it is the final
  submit action. Ask yourself: "Should the user be able to keep editing
  after this save?" If yes — don't close. Show a snackbar instead.
  Keep save and close as two separate concerns.


----------------------------------------------------------------
CHANGE 2 — saveInfraction and isCurrentTabValid used || instead of &&
FILE: demo-dialog.component.ts — saveInfraction(), isCurrentTabValid()
----------------------------------------------------------------

WHAT WAS WRONG:
  saveInfraction() checked:
    if (infractionForm.valid || infractionMaxForm.valid || ...)
  This meant it saved even if only ONE of the four sub-forms was valid,
  while the others still had errors. Partial/invalid data got saved.

  isCurrentTabValid() for Tab 6 had the same bug — it returned true
  (enabling the Save button) as soon as any single sub-form was valid.

WHAT WAS FIXED:
  Changed || to && in both places.
  Now ALL four infraction sub-forms must be valid before saving is allowed.

HOW TO AVOID IT NEXT TIME:
  When validating multiple forms that must ALL be valid to proceed, always
  use &&. Use || only when ANY ONE being valid is enough (rare for forms).
  Read the condition out loud: "save if form A is valid AND form B is valid"
  vs "save if form A is valid OR form B is valid." The first is almost
  always what you want.

CHANGE 3 — memory leaks from unmanaged subscriptions
FILE: demo-dialog.component.ts — openImportDialog(), openAddValueDialog()
----------------------------------------------------------------

WHAT WAS WRONG:
  Both methods called dialogRef.afterClosed().subscribe(...).
  These subscriptions were never cleaned up. Every time the user opened
  the import or add-value dialog, a new subscription was created and leaked.
  Over many opens, this accumulates handlers in memory that never get freed.

WHAT WAS FIXED:
  Added private destroy$ = new Subject<void>() to the component class.
  Added .pipe(takeUntil(this.destroy$)) to both subscriptions.
  Added ngOnDestroy() which calls destroy$.next() and destroy$.complete().
  This ensures all subscriptions are cleaned up when the dialog closes.

HOW TO AVOID IT NEXT TIME:
  Any time you call .subscribe() inside a component, ask: "Where does this
  get cleaned up?" The standard pattern in Angular is:
    private destroy$ = new Subject<void>();
    someObservable$.pipe(takeUntil(this.destroy$)).subscribe(...)
    ngOnDestroy() { this.destroy$.next(); this.destroy$.complete(); }
  Apply this to every component that has subscriptions.

CHANGE 5 — RaportDialogComponent: no validators, submits invalid data
FILE: raport-dialog.component.ts
----------------------------------------------------------------

WHAT WAS WRONG:
  All 5 form fields had no validators: baseReportName: [''], etc.
  onSubmit() called this.dialogRef.close(this.form.value) with no
  form.valid check — it submitted even when the form was empty.
  Also had a console.log(this.form.value) that would appear in production.

WHAT WAS FIXED:
  Added Validators.required to baseReportName.
  Fixed onSubmit() to check form.invalid first and call markAllAsTouched()
  to show errors if invalid.
  Removed console.log.
  Added MatSnackBar success feedback.

HOW TO AVOID IT NEXT TIME:
  Always add at minimum Validators.required to fields that must not be empty.
  Always guard onSubmit() with: if (this.form.invalid) { this.form.markAllAsTouched(); return; }
  Never leave console.log in production code — use proper logging or remove it.

CHANGE 8 — Math.random() used to generate record IDs
FILE: maintenance-dialog.component.ts — onSubmit()
----------------------------------------------------------------

WHAT WAS WRONG:
  id: Math.random().toString(36).substr(2, 9)
  Math.random() is not guaranteed to be unique — two quick saves could
  produce the same ID, causing one record to silently overwrite another.
  Also, substr() is deprecated in modern JavaScript.

WHAT WAS FIXED:
  Changed to: Date.now().toString(36) + Math.random().toString(36).substring(2, 7)
  Combining a timestamp (always increasing) with random characters makes
  collisions practically impossible.
  Also replaced .substr() with .substring() (the non-deprecated version).

HOW TO AVOID IT NEXT TIME:
  Never use Math.random() alone for IDs. Options from best to acceptable:
    1. Use crypto.randomUUID() (built into modern browsers, Angular 14+)
    2. Use a UUID library (uuid npm package)
    3. Use Date.now() + random suffix as done here
  In real apps, let the backend/database generate the ID.

CHANGE 9 — maintenance-dialog: data typed as 'any'
FILE: maintenance-dialog.component.ts — constructor
----------------------------------------------------------------

WHAT WAS WRONG:
  @Inject(MAT_DIALOG_DATA) public data: any
  Using 'any' removes all type safety. TypeScript cannot warn you if you
  access a property that doesn't exist on the data object.

WHAT WAS FIXED:
  Changed to: @Inject(MAT_DIALOG_DATA) public data: MaintenanceRecord | null
  Now TypeScript knows exactly what properties are available on data,
  and will error if you try to access something that doesn't exist.

HOW TO AVOID IT NEXT TIME:
  Never use 'any' unless absolutely necessary. For MAT_DIALOG_DATA,
  always type it as: YourInterface | null (null because the dialog can
  be opened without data for "create new" mode).

FILE: maintenance-dialog.component.html

WHAT WAS WRONG:

  The cost field had Validators.min(0) but no error message either.

    - conductor: "Conducteur requis"
    - type: "Type requis"
    - date: "Date requise"
    - cost: "Le coût ne peut pas être négatif"

HOW TO AVOID IT NEXT TIME:

  at the same time — don't add one without the other.

CHANGE 11 — maintenance-dialog: no isLoading state
FILE: maintenance-dialog.component.ts and .html
----------------------------------------------------------------

WHAT WAS WRONG:
  There was no isLoading flag. When connected to a real HTTP API,
  the user could click Save multiple times before the first request
  finished, causing duplicate submissions.

WHAT WAS FIXED:
  Added isLoading = false property to the component.
  Set isLoading = true before the save operation and false after.
  Added [disabled]="maintenanceForm.invalid || isLoading" to the Save button.

HOW TO AVOID IT NEXT TIME:
  Every form that will make a network request needs an isLoading flag.
  Always disable the submit button while a request is in flight.
  Pattern to follow:
    isLoading = false;
    onSubmit() {
      this.isLoading = true;
      this.service.save(data).subscribe({
        next: () => { ... },
        error: () => { ... },
        complete: () => { this.isLoading = false; }
      });
    }

CHANGE 12 — browser confirm() used for delete confirmation
FILE: vehicle-list.component.ts — deleteRecord()
----------------------------------------------------------------

WHAT WAS WRONG:
  if (confirm('Êtes-vous sûr...')) { ... }
  The browser's native confirm() dialog:
    - Looks ugly and inconsistent with Material Design
    - Blocks the JavaScript thread
    - Cannot be styled or customized
    - Ignored by some automated testing tools

WHAT WAS FIXED:
  Created a reusable ConfirmDialogComponent in shared/confirm-dialog.component.ts.
  deleteRecord() now opens this dialog and reacts to the result via observable.
  Also added takeUntil(destroy$) to prevent subscription leaks.

HOW TO AVOID IT NEXT TIME:
  Never use alert(), confirm(), or prompt() in Angular apps.
  Use MatDialog with a small confirmation component instead.
  The ConfirmDialogComponent in this project is reusable — pass any
  title and message via MAT_DIALOG_DATA.

CHANGE 15 — MatSnackBarModule and MatTooltipModule missing from app.module.ts
FILE: app.module.ts
----------------------------------------------------------------

WHAT WAS WRONG:
  MatSnackBar was injected into components (after our fixes) but
  MatSnackBarModule was not imported in AppModule — this would cause a
  runtime error when any snackbar was triggered.

  MatTooltip (matTooltip directive) was used in vehicle-list.component.html
  but MatTooltipModule was not imported — tooltips would silently not work.

WHAT WAS FIXED:
  Added MatSnackBarModule and MatTooltipModule to the imports array in AppModule.

HOW TO AVOID IT NEXT TIME:
  When you use any Angular Material component or directive, check that its
  module is in AppModule (or the feature module) imports array.
  Rule: one module per Material feature. Examples:
    mat-snack-bar    → MatSnackBarModule
    matTooltip       → MatTooltipModule
    mat-dialog       → MatDialogModule
  If it works in the template but shows nothing, a missing module is often why.
